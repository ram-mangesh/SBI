4.1 AUTHENTICATION ARCHITECTURE 4.1.1 LOGIN FLOW TypeScript

LOGIN FLOW SPECIFICATION
════════════════════════════════════════════════════════════════

Entry Points: ────────────

✅ Direct URL: /auth/login ✅ Redirect: From protected route (with
returnUrl) ✅ Session expired: Auto-redirect from any authenticated page
✅ Logout: User initiated logout

Backend API Mapping: ───────────────────

Endpoint: POST /auth/login Request Body: { “email”:
“priya.sharma@email.com”, “password”: “encrypted_password” }

Response (Success): { “status”: “SUCCESS”, “data”: { “token”:
“eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…”, “refreshToken”:
“refresh_token_string”, “customer”: { “id”: “CUST001”, “firstName”:
“Priya”, “lastName”: “Sharma”, “email”: “priya.sharma@email.com”,
“role”: “CUSTOMER”, “customerType”: “salaried”, “kycStatus”: “APPROVED”,
“profileCompletionPercentage”: 100 }, “expiresIn”: 86400, “requiresOTP”:
false }, “timestamp”: “2024-01-20T10:30:00Z” }

Response (OTP Required): { “status”: “OTP_REQUIRED”, “message”: “OTP
sent to registered mobile”, “data”: { “otpSentTo”: “+91-XXXXX-43210”,
“otpExpiresIn”: 300, “sessionId”: “temp_session_id” } }

Response (Error): { “status”: “ERROR”, “message”: “Invalid credentials”,
“errorCode”: “AUTH_001”, “timestamp”: “2024-01-20T10:30:00Z” }

Frontend Flow: ─────────────

Step 1: User Lands on Login Page ├─ Display: LoginForm component ├─
Pre-filled: Email if coming from registration ├─ State: IDLE └─ Focus:
Email input (auto-focus)

Step 2: User Enters Credentials ├─ Email validation: Real-time (on blur)
├─ Password validation: Minimum length check ├─ Show errors: Below
respective fields └─ Submit button: Disabled until valid

Step 3: User Submits Form ├─ State: SUBMITTING ├─ Button: Show spinner,
text “Logging in…” ├─ Disable: All form inputs ├─ API Call: POST
/auth/login └─ Timeout: 30 seconds

Step 4A: Success (No OTP) ├─ Receive: JWT token + customer data ├─ Store
Token: localStorage.setItem(‘authToken’, token) ├─ Store RefreshToken:
localStorage.setItem(‘refreshToken’, refreshToken) ├─ Store Customer:
Update AuthContext state ├─ Analytics: Track login event ├─ Redirect: │
├─ IF returnUrl exists → Navigate to returnUrl │ ├─ IF profile
incomplete → /customer/onboarding │ └─ ELSE → /customer/dashboard └─
Duration: 500ms delay for smooth transition

Step 4B: OTP Required ├─ Receive: OTP session ID ├─ Store:
sessionStorage.setItem(‘otpSessionId’, sessionId) ├─ Navigate:
/auth/otp-verify ├─ Pass: Phone number (masked) to OTP page └─ Start:
OTP expiry countdown (5 minutes)

Step 4C: Error ├─ State: ERROR ├─ Parse: Error message from response ├─
Display: Error alert above form ├─ Enable: All form inputs ├─ Focus:
Email or Password (based on error) ├─ Clear: Password field (security
best practice) └─ Log: Error to analytics

Login State Machine: ───────────────────

States: ├─ IDLE → Initial state ├─ VALIDATING → Client-side validation
in progress ├─ SUBMITTING → API call in progress ├─ OTP_REQUIRED → Need
OTP verification ├─ SUCCESS → Login successful, redirecting └─ ERROR →
Login failed, show error

Transitions: IDLE → VALIDATING (on form submit) VALIDATING → SUBMITTING
(validation passed) VALIDATING → ERROR (validation failed) SUBMITTING →
OTP_REQUIRED (OTP needed) SUBMITTING → SUCCESS (login successful)
SUBMITTING → ERROR (API error) OTP_REQUIRED → (navigate to OTP page)
SUCCESS → (redirect to destination) ERROR → IDLE (user retries)

Validation Rules: ────────────────

Email Field: ├─ Required: Yes ├─ Format: Valid email regex │ └─
/[1]+@[^\s@]+.[^\s@]+$/ ├─ Max length: 255 characters ├─ Trimmed:
Auto-trim whitespace └─ Error messages: ├─ Empty: “Email is required” ├─
Invalid: “Please enter a valid email” └─ Too long: “Email is too long”

Password Field: ├─ Required: Yes ├─ Min length: 8 characters (display
only, validated on backend) ├─ Max length: 128 characters ├─ Show/Hide:
Toggle button (eye icon) └─ Error messages: ├─ Empty: “Password is
required” └─ Too short: “Password must be at least 8 characters”

Form Validation Timing: ├─ Email: On blur (after user leaves field) ├─
Password: On submit (don’t reveal password requirements) ├─ Form: On
submit (all fields) └─ Re-validate: On field change after first submit

Security Features: ─────────────────

Password Security: ├─ Type: password (masked input) ├─ Autocomplete:
“current-password” ├─ Show/Hide toggle: Optional ├─ No copy/paste
restrictions (accessibility) └─ Clear on error: Yes (prevent resubmit of
wrong password)

CSRF Protection: ├─ CSRF token: Retrieved from backend meta tag ├─
Include in request: X-CSRF-TOKEN header └─ Validated: By Spring Security

Rate Limiting: ├─ Frontend: Max 3 attempts per minute ├─ Backend: Max 5
attempts per 5 minutes (IP-based) ├─ Show: Cooldown message if exceeded
└─ Unlock: After cooldown period

Brute Force Prevention: ├─ Failed attempts: Tracked in backend ├─
Account lock: After 5 failed attempts ├─ Lock duration: 15 minutes ├─
Message: “Account locked. Try after 15 minutes” └─ Alternative: “Forgot
password?” link

Session Hijacking Prevention: ├─ Token: Includes user agent hash ├─ IP
validation: Optional (backend) ├─ Fingerprinting: Device fingerprint
stored └─ Suspicious activity: Force re-login

Remember Me: ───────────

Checkbox: ├─ Label: “Remember me” ├─ Default: Unchecked ├─ Position:
Below password field └─ Tooltip: “Stay logged in for 30 days”

Behavior: IF checked: ├─ Set longer expiry: 30 days (backend) ├─ Store
preference: localStorage.setItem(‘rememberMe’, ‘true’) └─ Extended
session: Refresh token valid for 30 days

IF not checked: ├─ Session expiry: 24 hours (default) ├─ Session-only:
Clear on browser close (optional) └─ Standard refresh: 24-hour refresh
token

Social Login (Optional Future Feature):
───────────────────────────────────────

Providers: ├─ Google ├─ Facebook └─ Apple (iOS)

Implementation: ├─ OAuth 2.0 flow ├─ Redirect to provider ├─ Callback to
/auth/callback ├─ Exchange code for token └─ Create/link account

Display: ├─ Divider: “OR” ├─ Social buttons: Below login form ├─ Icons:
Provider logos └─ Text: “Continue with Google”

Analytics: ─────────

Track Events: ├─ login_page_viewed ├─ login_attempted ├─ login_success
├─ login_failed (with reason: invalid_credentials, account_locked, etc.)
├─ login_otp_required ├─ login_social_initiated (if applicable) └─
login_remember_me_used

Track Properties: ├─ login_method: “email” | “phone” | “social” ├─
error_code: If failed ├─ redirect_from: If redirected from protected
route ├─ device_type: “mobile” | “desktop” | “tablet” └─ browser: User
agent info

Error Handling: ──────────────

Network Errors: ├─ Timeout: After 30 seconds ├─ Offline: Check
navigator.onLine ├─ Retry: Show retry button └─ Message: “Network error.
Please check connection”

Server Errors: ├─ 500: “Server error. Please try again” ├─ 503: “Service
unavailable. Try later” └─ Fallback: “Something went wrong”

Validation Errors: ├─ 400: Parse field errors from response ├─ Display:
Below respective fields └─ Format: {field: errorMessage}

Authentication Errors: ├─ 401: “Invalid email or password” ├─ 403:
“Account suspended. Contact support” ├─ 429: “Too many attempts. Try
after X minutes” └─ Account locked: “Account locked. Reset password”

User Feedback: ─────────────

Loading State: ┌──────────────────────────────────┐ │ Email:
priya@email.com │ │ Password: ******** │ │ │ │ [⏳ Logging in…] │ │
(disabled, with spinner) │ └──────────────────────────────────┘

Success State: ┌──────────────────────────────────┐ │ ✓ Login
successful! │ │ Redirecting to dashboard… │
└──────────────────────────────────┘ (Brief message, then redirect)

Error State: ┌──────────────────────────────────┐ │ ✗ Invalid email or
password │ │ Please try again or reset │ │ your password. │
├──────────────────────────────────┤ │ Email: priya@email.com │ │
Password: [cleared] │ │ │ │ [Login] │

Backend API Mapping: ───────────────────

Endpoint: POST /customers/register Request Body: { “firstName”: “Priya”,
“lastName”: “Sharma”, “email”: “priya.sharma@email.com”, “phone”:
“+919876543210”, “password”: “encrypted_password”, “dob”: “1995-03-15”,
“employmentType”: “SALARIED”, “monthlyIncome”: 75000, “city”: “Mumbai”,
“agreeToTerms”: true }

Response (Success): { “status”: “SUCCESS”, “message”: “Registration
successful”, “data”: { “customerId”: “CUST001”, “email”:
“priya.sharma@email.com”, “phone”: “+919876543210”, “requiresOTP”: true,
“otpSentTo”: “+91-XXXXX-43210” }, “timestamp”: “2024-01-20T10:30:00Z” }

Response (Error - Duplicate): { “status”: “ERROR”, “message”: “Email
already registered”, “errorCode”: “REG_001”, “data”: { “field”: “email”,
“suggestion”: “Try logging in or reset password” } }

Frontend Flow: ─────────────

Step 1: User Lands on Registration Page ├─ Display: RegistrationForm
component ├─ Form Type: Multi-field form (all fields on one page) │ OR │
Multi-step wizard (Optional future enhancement) ├─ State: IDLE └─ Focus:
First name input

Step 2: User Fills Form ├─ Real-time validation: On blur for each field
├─ Password strength: Visual indicator ├─ Age calculation: Auto from DOB
├─ Show errors: Inline below fields └─ Submit button: Enabled when all
fields valid

Step 3: User Submits Form ├─ State: SUBMITTING ├─ Button: Show spinner,
text “Creating account…” ├─ Disable: All form inputs ├─ Hash password:
Client-side (optional, or send plain to HTTPS) ├─ API Call: POST
/customers/register └─ Timeout: 30 seconds

Step 4A: Success (OTP Required) ├─ Receive: Customer ID, OTP info ├─
Store: customerId in sessionStorage ├─ Analytics: Track registration
event ├─ Navigate: /auth/otp-verify ├─ Pass: Phone number (masked),
email └─ Message: “Account created! Verify your phone”

Step 4B: Success (No OTP) ├─ Receive: JWT token directly ├─ Store: Token
and customer data ├─ Navigate: /customer/onboarding └─ Message:
“Welcome! Complete your profile”

Step 4C: Error ├─ State: ERROR ├─ Parse: Field errors from response ├─
Display: Errors below respective fields ├─ Enable: All form inputs ├─
Focus: First error field └─ Log: Error to analytics

Registration Form Fields: ────────────────────────

Personal Information: ├─ First Name (required) ├─ Last Name (required)
├─ Email (required, unique) ├─ Phone (required, 10 digits, unique) ├─
Date of Birth (required, age >= 18) └─ Gender (optional)

Account Security: ├─ Password (required, min 8 chars) ├─ Confirm
Password (must match) └─ Show password toggle

Employment Information: ├─ Employment Type (required, dropdown) │
Options: Salaried, Self-Employed, Business, Freelancer, Student, Retired
├─ Monthly Income (required, number) └─ City (required, autocomplete)

Legal: ├─ Terms & Conditions checkbox (required) ├─ Privacy Policy
checkbox (required) └─ Marketing consent checkbox (optional)

Validation Rules: ────────────────

First Name / Last Name: ├─ Required: Yes ├─ Min length: 2 characters ├─
Max length: 50 characters ├─ Pattern: Letters only (Unicode support for
non-English) ├─ Trim: Auto-trim whitespace └─ Error: “Name must be at
least 2 characters”

Email: ├─ Required: Yes ├─ Format: Valid email regex ├─ Max length: 255
characters ├─ Uniqueness check: On blur (debounced API call) │ └─ GET
/auth/check-email?email=xxx └─ Errors: ├─ “Email is required” ├─
“Invalid email format” └─ “Email already registered” (with “Login” link)

Phone: ├─ Required: Yes ├─ Format: 10 digits (India) ├─ Pattern: /[2]$/
├─ Uniqueness check: On blur (debounced API call) │ └─ GET
/auth/check-phone?phone=xxx ├─ Auto-format: +91-XXXXX-XXXXX └─ Errors:
├─ “Phone is required” ├─ “Invalid phone number” └─ “Phone already
registered”

Date of Birth: ├─ Required: Yes ├─ Format: YYYY-MM-DD ├─ Age
validation: >= 18 years ├─ Max age: <= 120 years ├─ Date picker:
Calendar UI └─ Errors: ├─ “Date of birth is required” ├─ “You must be at
least 18 years old” └─ “Invalid date”

Password: ├─ Required: Yes ├─ Min length: 8 characters ├─ Strength
requirements (backend enforced, frontend hints): │ ├─ At least 1
uppercase letter │ ├─ At least 1 lowercase letter │ ├─ At least 1 number
│ └─ At least 1 special character ├─ Max length: 128 characters ├─
Strength indicator: Visual bar │ ├─ Weak: Red (< 50%) │ ├─ Medium:
Yellow (50-80%) │ └─ Strong: Green (> 80%) └─ Errors: ├─ “Password is
required” ├─ “Password must be at least 8 characters” └─ “Password is
too weak”

Confirm Password: ├─ Required: Yes ├─ Must match: Password field ├─
Real-time check: On change └─ Error: “Passwords do not match”

Employment Type: ├─ Required: Yes ├─ Options: Dropdown/Select ├─
Default: Not selected └─ Error: “Please select employment type”

Monthly Income: ├─ Required: Yes ├─ Type: Number (positive) ├─ Min: 0
(for students, can be 0) ├─ Max: 99,999,999 ├─ Format: Currency (₹
prefix) └─ Error: “Monthly income is required”

Terms & Conditions: ├─ Required: Yes (checkbox must be checked) ├─ Link:
Opens terms in new tab └─ Error: “You must agree to terms”

Password Strength Indicator: ────────────────────────────

Visual Representation: Weak: Password: ******** Strength: ████░░░░░░░░
Weak

Medium: Password: ******** Strength: ████████░░░░ Medium

Strong: Password: ******** Strength: ████████████ Strong ✓

Algorithm (Client-side estimation): score = 0 IF length >= 8: score +=
25 IF contains uppercase: score += 25 IF contains lowercase: score += 15
IF contains number: score += 20 IF contains special char: score += 15

Strength: ├─ 0-40: Weak ├─ 41-75: Medium └─ 76-100: Strong

Duplicate Check: ───────────────

Email Check: ├─ Trigger: On blur (after user leaves email field) ├─
Debounce: 500ms (prevent rapid API calls) ├─ API: GET
/auth/check-email?email=xxx ├─ Response (Available): │ └─ { “available”:
true } ├─ Response (Taken): │ └─ { “available”: false, “message”: “Email
already registered” } ├─ Display: Show error below email field └─
Suggest: “Already have an account? [Login]”

Phone Check: ├─ Similar to email check ├─ API: GET
/auth/check-phone?phone=xxx └─ Format: Auto-format before check
(+91-XXXXX-XXXXX)

Age Calculation: ───────────────

Auto-calculate age from DOB: dob = “1995-03-15” today = new Date() age =
today.year - dob.year

IF (today.month < dob.month) OR (today.month === dob.month AND today.day
< dob.day): age = age - 1

IF age < 18: Show error: “You must be at least 18 years old” Disable
submit button

FLOW TypeScript

OTP VERIFICATION FLOW SPECIFICATION
════════════════════════════════════════════════════════════════

Entry Points: ────────────

✅ After registration: /auth/otp-verify ✅ After login (if 2FA enabled):
/auth/otp-verify ✅ Phone verification: /auth/otp-verify

Backend API Mapping: ───────────────────

Endpoint: POST /auth/verify-otp Request Body: { “sessionId”:
“temp_session_id”, “otp”: “123456”, “verificationType”: “REGISTRATION” |
“LOGIN” | “PHONE_VERIFICATION” }

Response (Success): { “status”: “SUCCESS”, “message”: “OTP verified
successfully”, “data”: { “token”:
“eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9…”, “refreshToken”:
“refresh_token_string”, “customer”: { “id”: “CUST001”, “firstName”:
“Priya”, “email”: “priya@email.com”, “role”: “CUSTOMER”,
“profileCompletionPercentage”: 20 }, “expiresIn”: 86400 }, “timestamp”:
“2024-01-20T10:30:00Z” }

Response (Error - Invalid OTP): { “status”: “ERROR”, “message”: “Invalid
OTP”, “errorCode”: “OTP_001”, “data”: { “attemptsRemaining”: 2,
“maxAttempts”: 3 } }

Response (Error - Expired): { “status”: “ERROR”, “message”: “OTP
expired”, “errorCode”: “OTP_002”, “data”: { “canResend”: true } }

Resend OTP API: Endpoint: POST /auth/resend-otp Request Body: {
“sessionId”: “temp_session_id”, “channel”: “SMS” | “EMAIL” }

Response: { “status”: “SUCCESS”, “message”: “OTP sent successfully”,
“data”: { “otpSentTo”: “+91-XXXXX-43210”, “otpExpiresIn”: 300,
“canResendAfter”: 60 } }

Frontend Flow: ─────────────

Step 1: User Lands on OTP Page ├─ Receive: sessionId, phone/email from
previous page ├─ Display: OTP input (6 digits) ├─ Show: Masked contact
(e.g., +91-XXXXX-43210) ├─ Start: Countdown timer (5 minutes) ├─ State:
IDLE └─ Focus: First OTP input box

Step 2: User Enters OTP ├─ Input: 6-digit code ├─ Auto-advance: After
each digit entered ├─ Auto-submit: When 6 digits entered (optional) ├─
Show: Loading state on submit └─ Validation: Client-side (6 digits,
numeric)

Step 3: User Submits OTP ├─ State: VERIFYING ├─ Button: Show spinner,
text “Verifying…” ├─ Disable: OTP inputs ├─ API Call: POST
/auth/verify-otp └─ Timeout: 15 seconds

Step 4A: Success ├─ Receive: JWT token + customer data ├─ Store: Token,
refresh token, customer data ├─ Analytics: Track OTP verification
success ├─ Navigate: │ ├─ IF registration → /customer/onboarding │ ├─ IF
login → /customer/dashboard │ └─ IF phone verification → Return to
settings └─ Show: Brief success message before redirect

Step 4B: Error (Invalid OTP) ├─ State: ERROR ├─ Parse: Error message,
attempts remaining ├─ Display: Error below OTP input ├─ Clear: OTP input
├─ Focus: First OTP input ├─ Show: Attempts remaining (e.g., “2 attempts
left”) └─ If max attempts: Lock and show “Request new OTP”

Step 4C: Error (Expired OTP) ├─ State: EXPIRED ├─ Disable: OTP input ├─
Show: “OTP expired” message ├─ Enable: Resend button └─ Clear: Timer

Step 5: Resend OTP ├─ User clicks: “Resend OTP” button ├─ API Call: POST
/auth/resend-otp ├─ Success: │ ├─ Show: “OTP sent” message │ ├─ Restart:
5-minute timer │ ├─ Start: 60-second cooldown (can’t resend again) │ └─
Clear: OTP input └─ Error: Show error message

OTP Input Design: ────────────────

6 Separate Input Boxes: ┌───┬───┬───┬───┬───┬───┐ │ 1 │ 2 │ 3 │ 4 │ 5 │
6 │ └───┴───┴───┴───┴───┴───┘

Behavior: ├─ Auto-focus next box: On digit entry ├─ Auto-focus previous:
On backspace (if current empty) ├─ Paste support: Paste 6 digits fills
all boxes ├─ Number only: inputMode=“numeric”, pattern=“[0-9]” └─
Auto-submit: When all 6 filled (optional)

Alternative: Single Input ┌─────────────┐ │ 1 2 3 4 5 6 │
└─────────────┘ (Visually separated, but single input)

Countdown Timer: ───────────────

Display: “Code expires in 04:32” (MM:SS format)

States: ├─ > 2 minutes: Green text ├─ 1-2 minutes: Yellow text ├─ < 1
minute: Red text, flashing └─ Expired: “Code expired” (red)

Implementation: startTime = 300 seconds (5 minutes) setInterval:
currentTime = currentTime - 1 IF currentTime <= 0: state = EXPIRED
clearInterval disable OTP input enable resend button

Resend Cooldown: resendCooldown = 60 seconds After resend: Start
cooldown timer Disable resend button Show: “Resend in 00:XX” After
cooldown: Enable resend button

Attempts Tracking: ─────────────────

Max Attempts: 3 (backend enforced)

Display: ┌──────────────────────────────────┐ │ ✗ Invalid OTP │ │ 2
attempts remaining │ └──────────────────────────────────┘

After 3 Failed Attempts: ┌──────────────────────────────────┐ │ ✗
Maximum attempts exceeded │ │ Please request a new OTP │ │ [Request New
OTP] │ └──────────────────────────────────┘

Security lockout: Backend locks session New OTP required: User must
request new OTP

OTP Validation: ──────────────

Client-Side: ├─ Length: Exactly 6 digits ├─ Type: Numeric only (0-9) ├─
Required: All 6 boxes filled └─ Format: No spaces, no letters

Backend: ├─ Validity: 5 minutes from send ├─ Max attempts: 3 per OTP ├─
Session validation: sessionId must match └─ One-time use: OTP
invalidated after verification

Channel Selection: ─────────────────

If both email and phone available: ┌──────────────────────────────────┐
│ Verify your account │ │ │ │ OTP sent to: │ │ ○ SMS: +91-XXXXX-43210 │
│ ○ Email: pr***@email.com │ │ │ │ [Switch to Email] │
└──────────────────────────────────┘

Switch: ├─ User clicks: “Switch to Email” ├─ API Call: POST
/auth/resend-otp (channel: EMAIL) ├─ Update: Display and timer └─ Clear:
OTP input

Track Events: ├─ otp_page_viewed ├─ otp_entered (partial, 3/6 digits) ├─

Session Expiry States: ─────────────────────

States: ├─ ACTIVE: Token valid, user active ├─ IDLE: Token valid, user
inactive ├─ EXPIRING_SOON: < 5 minutes until token expiry ├─ WARNING: <
2 minutes until inactivity logout ├─ EXPIRED: Token expired └─
TERMINATED: Session forcefully terminated

Transitions: ACTIVE → IDLE (no activity for X minutes) IDLE → ACTIVE
(user interaction) ACTIVE → EXPIRING_SOON (token nearing expiry)
EXPIRING_SOON → ACTIVE (token refreshed) IDLE → WARNING (nearing
inactivity timeout) WARNING → ACTIVE (user clicks stay logged in)
WARNING → TERMINATED (countdown reaches 0) EXPIRED → TERMINATED (token

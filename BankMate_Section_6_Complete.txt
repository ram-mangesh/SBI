6.1 DASHBOARD ARCHITECTURE OVERVIEW
Purpose
The BankMate AI Dashboard is the primary post-authentication interface for all user roles. It serves as the unified command center that:
PRIMARY OBJECTIVES
════════════════════════════════════════════════════════════════

✅ Surface AI-generated hyper-personalized product recommendations
✅ Display life-event-driven financial insights
✅ Drive product conversion through contextual CTAs
✅ Track customer financial goals and progress
✅ Communicate via all channels (push, SMS, email, WhatsApp, voice)
✅ Present onboarding completion nudges
✅ Show real-time application status
✅ Enable digital banking adoption through guided quick actions

Dashboard Principles
DESIGN PRINCIPLES
════════════════════════════════════════════════════════════════

PRINCIPLE 1: AI-First Surface
──────────────────────────────
Every widget is AI-aware.
Recommendations, insights, and nudges are dynamically injected
from existing AI agents via backend APIs.
No static content on the dashboard.

PRINCIPLE 2: Life-Event Driven
────────────────────────────────
Dashboard layout and widget priority shift based on detected
life events. A customer with a detected "Home Purchase" event
sees home loan widgets promoted. A "New Baby" event promotes
child plan insurance.

PRINCIPLE 3: Customer Type Aware
──────────────────────────────────
Widget configuration differs per customer type:
- Salaried: EMI eligibility, salary account upsell
- Self-Employed: Business loan, overdraft
- Student: Education loan, credit card starter
- Senior: FD rates, health insurance, pension
- NRI: Remittance, NRE/NRO account

PRINCIPLE 4: Conversion Optimized
────────────────────────────────────
Every recommendation card has exactly ONE primary CTA.
CTAs map to existing application journey routes.
No dead ends. Every widget leads forward.

PRINCIPLE 5: Progressive Disclosure
──────────────────────────────────────
Dashboard surfaces summary-level data.
Drill-down navigates to dedicated feature pages.
This preserves performance and reduces cognitive load.

PRINCIPLE 6: Offline Resilient
────────────────────────────────
Critical widgets use Redis-cached data.
Stale-while-revalidate pattern for all dashboard API calls.
Offline indicator shown when connectivity is lost.

6.2.1 Welcome Area
WELCOME AREA SPECIFICATION
════════════════════════════════════════════════════════════════

Component:      WelcomeWidget
Position:       Top of main content, full width
Priority:       P0 (Always visible)

────────────────────────────────────────────────────────────────
DATA BINDING
────────────────────────────────────────────────────────────────

API:            GET /customers/{id}
Cache:          Redis key: customer:{id}:profile (TTL: 5m)
Store:          customer.store.ts → customerProfile

Fields Used:
  ├── firstName               → Greeting text
  ├── customerType            → Type badge
  ├── profileCompletionPct    → Progress bar (if < 100%)
  ├── kycStatus               → KYC nudge (if PENDING)
  ├── lastLoginAt             → "Last seen" info
  └── preferredLanguage       → i18n locale

────────────────────────────────────────────────────────────────
DISPLAY LOGIC
────────────────────────────────────────────────────────────────

Greeting Logic:
  IF currentHour >= 5 AND < 12  → "Good Morning, {firstName}!"
  IF currentHour >= 12 AND < 17 → "Good Afternoon, {firstName}!"
  IF currentHour >= 17 AND < 21 → "Good Evening, {firstName}!"
  ELSE                           → "Welcome back, {firstName}!"

Customer Type Badge:
  ├── SALARIED      → Badge: "Salaried Professional"  Color: Blue
  ├── SELF_EMPLOYED → Badge: "Business Owner"         Color: Orange
  ├── STUDENT       → Badge: "Student"                Color: Green
  ├── SENIOR        → Badge: "Senior Citizen"         Color: Purple
  └── NRI           → Badge: "NRI Customer"           Color: Teal

Profile Completion Nudge:
  Condition: profileCompletionPct < 100
  Shows:
    ├── Progress bar (e.g., "70% Complete")
    ├── Text: "Complete your profile to unlock personalised offers"
    └── CTA: "Complete Profile" → /customer/onboarding/:nextStep

KYC Nudge:
  Condition: kycStatus = PENDING or NOT_SUBMITTED
  Shows:
    ├── Warning banner
    ├── Text: "Complete KYC to apply for products"
    └── CTA: "Start KYC" → /customer/kyc

────────────────────────────────────────────────────────────────
AI INTEGRATION
────────────────────────────────────────────────────────────────

Agent:   NextBestActionAgent
Trigger: On dashboard load
Output:  Dynamic welcome message injected below greeting
Example: "You have 3 new personalised product recommendations
          based on your recent activity."

────────────────────────────────────────────────────────────────
KAFKA EVENTS CONSUMED
────────────────────────────────────────────────────────────────

Event:  CUSTOMER_PROFILE_UPDATED
Action: Refresh customerProfile from Redis/API
Event:  KYC_STATUS_UPDATED
Action: Re-evaluate KYC nudge visibility

  ├── Label: "Monthly Income"
  ├── Value: ₹{monthlyIncome} (formatted via currency.utils.ts)
  └── Trend: Compared to last month (if available)

  ├── Label: "Monthly Expenses"
  ├── Value: ₹{monthlyExpenses}
  └── Visual: Horizontal bar (expenses/income ratio)

Available Income Card:
  ├── Label: "Available Balance"
  ├── Value: ₹{availableIncome}
  └── Context: "After EMIs and expenses"

Credit Health Indicator:
  ├── Condition: creditScore is set
  ├── Score range:
  │     300–549 → Poor    (Red)
  │     550–649 → Fair    (Orange)
  │     650–749 → Good    (Yellow)
  │     750–849 → Great   (Light Green)
  │     850–900 → Excellent (Green)
  └── CTA: "Improve Score" → /customer/profile/financial

EMI Load Meter:
  ├── Formula: (existingEMI / monthlyIncome) × 100
  ├── 0–30%   → Healthy (Green)
  ├── 31–50%  → Moderate (Yellow)
  └── 51%+    → High (Red)

────────────────────────────────────────────────────────────────
NAVIGATION
────────────────────────────────────────────────────────────────

"View Details" CTA → /customer/profile/financial
"Full Report" CTA  → /customer/transactions

Action: Invalidate Redis cache, refresh widget, show toast

Action: Update stage badge without full refresh

6.2.4 AI Recommendation Center (Primary Conversion Widget)
AI RECOMMENDATION CENTER SPECIFICATION
════════════════════════════════════════════════════════════════

Component:      AIRecommendationCenterWidget
Position:       Row 3, Full width
Priority:       P0 (Core conversion widget)

This is the PRIMARY product conversion surface on the dashboard.
All recommendations originate from the existing
RecommendationEngine AI Agent.

────────────────────────────────────────────────────────────────
DATA BINDING
────────────────────────────────────────────────────────────────

API:            GET /recommendations/{customerId}
Cache:          Redis key: recommendations:{customerId} (TTL: 10m)
Store:          recommendations.store.ts → activeRecommendations

Response Fields Used:
  ├── recommendationId        → Unique card key
  ├── productId               → Link to product detail
  ├── productName             → Card title
  ├── productCategory         → Category badge
  ├── matchScore              → Match percentage display
  ├── affordabilityStatus     → Affordability indicator
  ├── eligibilityStatus       → Eligibility indicator
  ├── confidenceScore         → AI confidence display
  ├── reasoningFactors[]      → "Why this product" section
  ├── suggestedActions[]      → Action chips
  ├── lifeEventId             → If life-event-triggered
  └── expiresAt               → Urgency indicator

────────────────────────────────────────────────────────────────
WIDGET LAYOUT
────────────────────────────────────────────────────────────────

Header Row:
  ├── Title: "Recommended For You"
  ├── AI Badge: "Powered by BankMate AI"
  ├── Filter: [All | Loans | Cards | Insurance | Investments]
  └── "View All" CTA → /customer/recommendations

Recommendation Cards (Horizontal Scroll on Mobile):
  Maximum displayed: 4 cards on desktop, 1 card on mobile
  Layout: Horizontal card row with scroll

────────────────────────────────────────────────────────────────
RECOMMENDATION CARD STRUCTURE
────────────────────────────────────────────────────────────────

Each card contains:

  Top Section:
  ├── Product Category Icon
  ├── Product Name
  ├── Category Badge (Loan / Card / Insurance / Investment)
  └── Life Event Tag (if lifeEventId is set)
      Example: "Based on your Home Purchase plan"

  Score Section:
  ├── Match Score:
  │     Display: "{matchScore}% Match"
  │     Visual: Circular progress indicator
  │     Color:
  │       90-100% → Green
  │       70-89%  → Blue
  │       50-69%  → Orange
  │       < 50%   → Grey (deprioritized)
  │
  ├── Affordability:
  │     AFFORDABLE      → ✅ "Within your budget"
  │     BORDERLINE      → ⚠️ "Near your budget limit"
  │     NOT_AFFORDABLE  → ❌ "Outside current budget"
  │
  └── Eligibility:
        ELIGIBLE        → ✅ "You're eligible"
        LIKELY_ELIGIBLE → 🔵 "Likely eligible"
        NOT_ELIGIBLE    → ❌ "Currently ineligible"

  Reasoning Section:
  ├── Title: "Why we recommend this"
  └── Pills: [{reasoningFactor}] (max 3 visible)
      Examples:
        "Matches your savings goal"
        "Suits your income bracket"
        "Popular with your peer group"
        "Linked to your life event"

  Action Section:
  ├── Primary CTA: "Apply Now" → /customer/applications/new?productId={id}
  │     Condition: eligibilityStatus = ELIGIBLE
  │
  ├── Primary CTA: "Check Eligibility" → /customer/products/{productId}
  │     Condition: eligibilityStatus = LIKELY_ELIGIBLE
  │
  ├── Primary CTA: "View Details" → /customer/products/{productId}
  │     Condition: NOT_ELIGIBLE
  │
  └── Secondary CTA: "Not Interested" → POST /recommendations/{id}/dismiss

────────────────────────────────────────────────────────────────
AI AGENT INTEGRATION
────────────────────────────────────────────────────────────────

Agent:          RecommendationEngine
Trigger:        GET /recommendations/{customerId}
Inputs:
  ├── customerType
  ├── financialProfile
  ├── activeLifeEvents
  ├── interactionHistory
  └── productCatalog

Output:         Ranked recommendation list with scores and reasons

Cache Strategy: Redis stores recommendation response for 10 minutes.
                On dismissal, local state updated immediately.
                Redis invalidated via Kafka: RECOMMENDATION_UPDATED

────────────────────────────────────────────────────────────────
KAFKA EVENTS
────────────────────────────────────────────────────────────────

Consumed:
  ├── RECOMMENDATION_UPDATED   → Refresh recommendations widget
  └── LIFE_EVENT_DETECTED      → Trigger recommendation refresh

Produced (Frontend → Backend via API):
  ├── POST /recommendations/{id}/view    → On card visible
  └── POST /recommendations/{id}/dismiss → On "Not Interested"

6.3 DASHBOARD WIDGET ARCHITECTURE
Reusable Widget Standards
WIDGET STANDARD SPECIFICATION
════════════════════════════════════════════════════════════════

Every dashboard widget MUST implement the following interface:

Widget Contract:
───────────────
  Props:
    ├── widgetId: string           → Unique identifier
    ├── title: string              → Display title
    ├── isLoading: boolean         → Loading state flag
    ├── error: Error | null        → Error state
    ├── isEmpty: boolean           → Empty state flag
    ├── lastRefreshedAt: Date      → Cache freshness indicator
    ├── onRefresh: () => void      → Manual refresh handler
    └── permissions: string[]      → Required permissions

  Internal States:
    ├── LOADING   → Skeleton UI shown
    ├── ERROR     → Error state UI shown
    ├── EMPTY     → Empty state UI shown
    ├── STALE     → Data shown with "Refresh" indicator
    └── READY     → Normal data display

Widget Header (standard across all widgets):
  ├── Widget Title
  ├── Last refreshed timestamp (optional, on hover)
  ├── Refresh icon button (on hover)
  └── "View All" / navigation CTA (where applicable)

Widget Container:
  ├── Background: Card surface (theme-aware)
  ├── Border radius: 12px
  ├── Shadow: elevation-1 (from Section 3 theme)
  ├── Padding: 20px
  └── Minimum height: 120px (prevents layout shift)

6.3.1 Widget Loading State
WIDGET LOADING STATE
════════════════════════════════════════════════════════════════

Trigger:  isLoading = true

Display:
  ├── Widget header: Skeleton bar (80% width)
  ├── Content area: Skeleton blocks matching widget shape
  └── No spinner overlay (skeleton preferred per UX principle)

Skeleton Shapes:
  ├── Financial Snapshot  → 3 skeleton metric cards
  ├── Recommendation      → 2 skeleton product cards
  ├── Life Event Banner   → 1 skeleton event card
  ├── Goal Tracker        → 2 skeleton goal rows
  ├── Recent Activity     → 5 skeleton transaction rows
  └── Notifications       → 4 skeleton notification rows

Timing:
  ├── Show skeleton immediately on mount
  ├── Minimum display: 300ms (prevents flash)
  └── Transition: Fade-out skeleton, fade-in content

6.3.2 Widget Error State
WIDGET ERROR STATE
════════════════════════════════════════════════════════════════

Trigger:  error !== null

Display:
  ├── Error icon: ⚠️
  ├── Error message: Human-readable (not raw API error)
  ├── Retry button: "Try Again" → calls onRefresh()
  └── Support link: "Contact Support" → /help

Error Message Mapping:
  ├── 401   → "Session expired. Please log in again."
  ├── 403   → "You don't have permission to view this."
  ├── 404   → "Data not found."
  ├── 429   → "Too many requests. Please wait a moment."
  ├── 500   → "Something went wrong on our end. Retrying..."
  └── Network → "No internet connection detected."

Auto-Retry Logic:
  ├── 500 errors: Auto-retry after 5s, max 3 attempts
  ├── Network errors: Retry on reconnection (event listener)
  └── 4xx errors: No auto-retry, show manual retry

6.3.4 Widget Refresh Behaviour
WIDGET REFRESH SPECIFICATION
════════════════════════════════════════════════════════════════

Refresh Triggers:
  ├── Manual: User clicks refresh icon
  ├── Auto:   TTL-based (per widget cache TTL)
  ├── Event:  Kafka event consumed (widget-specific)
  └── Focus:  App returns to foreground (visibilitychange)

Stale Data Indicator:
  Condition: (currentTime - lastRefreshedAt) > widgetTTL
  Display:   "Last updated {X} minutes ago" below widget title
  Behaviour: Data remains visible (stale-while-revalidate)

Background Refresh:
  ├── Widget does NOT show loading skeleton on background refresh
  ├── Data updates silently
  └── New data fades in (opacity transition)

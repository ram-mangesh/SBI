🎯 SECTION OVERVIEW

════════════════════════════════════════════════════════════════
Section 8 defines the complete frontend specification for the

BankMate AI product recommendation and product discovery experience.
This section covers everything a customer sees, interacts with, and

navigates through when discovering, evaluating, comparing, and

applying for banking products — from the AI-powered recommendation

feed to the product detail journey to the final application CTA.
All recommendations are sourced exclusively from the existing

Recommendation Engine. All eligibility checks use the existing

QualificationAgent. All product data uses existing product APIs.

No new backend components are introduced.
Key Integration Points:

RecommendationEngine    → Scores and ranks products per customer
QualificationAgent      → Real-time eligibility evaluation
LifeEventDetector       → Life event context for recommendation ranking
FinancialAnalysisAgent  → Affordability computation
NextBestActionAgent     → CTA priority per recommendation card
ConversationalAgent     → AI explanation of recommendations
Kafka Events            → Product interaction tracking
Redis Cache             → Recommendation list caching
Existing APIs           → All product, recommendation, application endpoints

8.1 RECOMMENDATION ARCHITECTURE

════════════════════════════════════════════════════════════════
8.1.1 ARCHITECTURE OVERVIEW
RECOMMENDATION SYSTEM — FRONTEND LAYER

════════════════════════════════════════════════════════════════
The frontend recommendation experience operates as a pure consumer

of the backend Recommendation Engine. The frontend:
✅ Fetches ranked recommendations via existing API

✅ Renders match scores, affordability signals, and life event context

✅ Presents AI explanations sourced from ConversationalAgent

✅ Triggers eligibility checks via QualificationAgent API

✅ Routes all product interactions to existing application journey

✅ Fires Kafka-consumed analytics events per interaction

✅ Reads from Redis-cached recommendation lists for performance
❌ Does NOT rank or score products on the frontend

❌ Does NOT compute eligibility on the frontend

❌ Does NOT generate AI explanations on the frontend

❌ Does NOT modify product data

❌ Does NOT introduce new scoring logic

8.1.5 CACHE STRATEGY
REDIS CACHE REFERENCE (EXISTING STRATEGY)

════════════════════════════════════════════════════════════════
Cache Keys (read by frontend, managed by backend):

├── recommendations:{customerId}           TTL: 30 min

├── products:catalog                       TTL: 60 min

├── products:{productId}                   TTL: 60 min

└── eligibility:{customerId}:{productId}   TTL: 15 min
Frontend Cache Behaviour:

├── On page load: Check store lastFetched timestamp

├── If < 5 minutes old: Render from store (no API call)

├── If 5–30 minutes old: Render stale + revalidate in background

├── If > 30 minutes: Show loader, fetch fresh

└── On profile update: Invalidate store, force fresh fetch

8.13 PRODUCT CTA BEHAVIOUR

════════════════════════════════════════════════════════════════
8.13.1 APPLY NOW CTA FLOW
"Apply Now" Button — Complete Decision Tree:
Step 1: Authentication Check

└── If not authenticated → Redirect to /auth/login?returnUrl={currentPath}
Step 2: KYC Status Check (from auth store, Section 4)

├── KYC status: APPROVED → Proceed to Step 3

├── KYC status: PENDING → Show modal: "Your KYC is under review. We'll notify you when approved."

├── KYC status: REJECTED → Show modal: "KYC rejected. Please re-upload documents." [Go to KYC]

└── KYC status: NOT_STARTED → Show modal: "Complete KYC to apply" [Start KYC] → /customer/kyc?returnUrl={applyUrl}
Step 3: Eligibility Pre-check

├── If eligibilityStatus === LIKELY_ELIGIBLE → Proceed directly to application

├── If eligibilityStatus === CHECK_ELIGIBILITY → Show eligibility check (8.8) first

│    Customer can choose to continue after seeing results

├── If eligibilityStatus === MAY_NOT_QUALIFY → Show warning modal:

│    "You may not fully qualify. Applying may result in rejection.

│     Do you still want to proceed?"

│    [Proceed Anyway] [See Alternatives]

└── If eligibilityStatus === NOT_ELIGIBLE → Button is disabled (8.4.4)
Step 4: Navigate to Application Page

└── Navigate to: /customer/products/{productId}/apply

├── Route defined in Section 2

├── Carries: productId, recId (if from recommendation)

└── Application flow continues in Section 9 (upcoming)
8.13.2 SAVE CTA BEHAVIOUR
├── API: PUT /recommendations/{recId}/save

├── Optimistic UI: Toggle saved state immediately

├── Toast (success): "Saved to your list" + "View saved →" action

├── Toast (unsave): "Removed from saved"

├── Error: Toast "Could not save. Please try again." + revert state

└── Saved items accessible at: /customer/recommendations?tab=saved
8.13.3 COMPARE CTA BEHAVIOUR
├── First product added: Show sticky comparison bar (8.6.2)

├── Second product added: Enable "Compare Now" button

├── Third product added: Disable additional compare buttons on other cards

│    with tooltip: "Remove a product to add another"

├── "Compare Now": Navigate to /customer/recommendations/compare

└── Remove from bar: Remove product from comparison store
8.13.4 DISMISS CTA BEHAVIOUR
Dismiss is accessible via:

├── Kebab menu (⋮) on recommendation card → "Not interested"

├── Swipe left on mobile (optional gesture)
Dismiss Flow:

├── API: PUT /recommendations/{recId}/dismiss

├── Optimistic UI: Animate card out of grid

├── Undo Toast: "Product removed" + [Undo] link (5 second window)

├── Undo action: PUT /recommendations/{recId}/restore

└── After undo window: Dismissal is permanent in current session

(Backend may re-surface after recommendation refresh cycle)

8.23 MERMAID RECOMMENDATION FLOW

════════════════════════════════════════════════════════════════
graph TB
    Start([Customer opens Recommendations]) --> FetchRec[GET /recommendations/{customerId}]

    FetchRec --> CacheCheck{Redis Cache\nFresh?}

    EngineProcess --> QualAgent[QualificationAgent\nPre-evaluates Eligibility]
    QualAgent --> FinAgent[FinancialAnalysisAgent\nAffordability Score]
    FinAgent --> ReturnList[Return Ranked Rec List]
    ReturnList --> RenderGrid[Render Recommendation Grid]
    RenderFromStore --> RenderGrid

    RenderGrid --> LifeEventCheck{Life Event\nDetected?}
    LifeEventCheck -->|Yes| ShowBanner[Show Life Event\nContext Banner]
    LifeEventCheck -->|No| HideBanner[No Banner]

    ShowBanner --> CardInteraction
